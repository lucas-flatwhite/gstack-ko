# /retro — 주간 엔지니어링 회고

커밋 히스토리, 작업 패턴, 코드 품질 메트릭을 분석하는 포괄적인 엔지니어링 회고를 생성합니다. 팀 인식: 커맨드를 실행하는 사용자를 식별하고, Claude Code를 힘 증폭기로 사용하는 시니어 IC/CTO 레벨의 빌더를 위해 설계된 개인별 칭찬과 성장 기회와 함께 모든 기여자를 분석합니다.

## 사용자 호출 가능
사용자가 `/retro`를 입력하면 이 스킬을 실행합니다.

## 인수
- `/retro` — 기본값: 지난 7일
- `/retro 24h` — 지난 24시간
- `/retro 14d` — 지난 14일
- `/retro 30d` — 지난 30일
- `/retro compare` — 현재 기간 vs 동일 길이의 이전 기간 비교
- `/retro compare 14d` — 명시적 기간으로 비교

## 지침

인수를 파싱하여 시간 기간을 결정합니다. 인수가 없으면 기본값 7일. git log 쿼리에 `--since="N days ago"`, `--since="N hours ago"`, 또는 `--since="N weeks ago"` (`w` 단위)를 사용합니다. 모든 시간은 **태평양 시간**으로 보고합니다 (타임스탬프 변환 시 `TZ=America/Los_Angeles` 사용).

**인수 검증:** 인수가 숫자 + `d`, `h`, `w`, 단어 `compare`, 또는 숫자 + `d`/`h`/`w`가 뒤따르는 `compare`와 일치하지 않으면, 다음 사용법을 보여주고 중지합니다:
```
사용법: /retro [기간]
  /retro              — 지난 7일 (기본값)
  /retro 24h          — 지난 24시간
  /retro 14d          — 지난 14일
  /retro 30d          — 지난 30일
  /retro compare      — 이번 기간 vs 이전 기간 비교
  /retro compare 14d  — 명시적 기간으로 비교
```

### Step 1: 원시 데이터 수집

먼저 origin을 가져오고 현재 사용자를 식별합니다:
```bash
git fetch origin main --quiet
# 회고를 실행하는 사람 식별
git config user.name
git config user.email
```

`git config user.name`이 반환하는 이름이 **"당신"** — 이 회고를 읽는 사람입니다. 다른 모든 작성자는 팀원입니다. 내러티브 방향 설정: "당신의" 커밋 vs 팀원 기여.

다음 모든 git 커맨드를 병렬로 실행합니다 (독립적):

```bash
# 1. 타임스탬프, 제목, 해시, 작성자, 변경된 파일, 삽입, 삭제를 포함한 기간 내 모든 커밋
git log origin/main --since="<window>" --format="%H|%aN|%ae|%ai|%s" --shortstat

# 2. 작성자와 함께 커밋당 테스트 vs 전체 LOC 분석
git log origin/main --since="<window>" --format="COMMIT:%H|%aN" --numstat

# 3. 세션 감지 및 시간별 분포를 위한 커밋 타임스탬프 (작성자 포함)
TZ=America/Los_Angeles git log origin/main --since="<window>" --format="%at|%aN|%ai|%s" | sort -n

# 4. 가장 자주 변경된 파일 (핫스팟 분석)
git log origin/main --since="<window>" --format="" --name-only | grep -v '^$' | sort | uniq -c | sort -rn

# 5. 커밋 메시지에서 PR 번호 추출 (#NNN 패턴)
git log origin/main --since="<window>" --format="%s" | grep -oE '#[0-9]+' | sed 's/^#//' | sort -n | uniq | sed 's/^/#/'

# 6. 작성자별 파일 핫스팟 (누가 무엇을 건드리는지)
git log origin/main --since="<window>" --format="AUTHOR:%aN" --name-only

# 7. 작성자별 커밋 카운트 (빠른 요약)
git shortlog origin/main --since="<window>" -sn --no-merges
```

### Step 2: 메트릭 계산

다음 메트릭을 요약 테이블에 계산하고 제시합니다:

| 메트릭 | 값 |
|--------|---|
| main에 대한 커밋 | N |
| 기여자 | N |
| 병합된 PR | N |
| 총 삽입 | N |
| 총 삭제 | N |
| 순 LOC 추가 | N |
| 테스트 LOC (삽입) | N |
| 테스트 LOC 비율 | N% |
| 버전 범위 | vX.Y.Z.W → vX.Y.Z.W |
| 활동 일수 | N |
| 감지된 세션 | N |
| 평균 LOC/세션-시간 | N |

그런 다음 바로 아래에 **작성자별 리더보드** 표시:

```
기여자              커밋   +/-          주요 영역
나 (garry)            32   +2400/-300   browse/
alice                 12   +800/-150    app/services/
bob                    3   +120/-40     tests/
```

커밋 내림차순으로 정렬합니다. 현재 사용자(`git config user.name`)는 항상 "나 (이름)"으로 레이블 지정되어 첫 번째로 나타납니다.

### Step 3: 커밋 시간 분포

태평양 시간으로 시간별 히스토그램을 바 차트로 표시합니다:

```
시간  커밋  ████████████████
 00:    4   ████
 07:    5   █████
 ...
```

다음을 확인하고 언급합니다:
- 피크 시간
- 활동 없는 시간대
- 패턴이 이중 모드(아침/저녁)인지 연속인지
- 야간 코딩 클러스터 (오후 10시 이후)

### Step 4: 작업 세션 감지

연속 커밋 간 **45분 갭** 임계값을 사용하여 세션을 감지합니다. 각 세션에 대해 보고:
- 시작/종료 시간 (태평양)
- 커밋 수
- 분 단위 소요 시간

세션 분류:
- **딥 세션** (50분 이상)
- **중간 세션** (20-50분)
- **마이크로 세션** (<20분, 일반적으로 단일 커밋 빠른 처리)

계산:
- 총 활성 코딩 시간 (세션 소요 시간 합계)
- 평균 세션 길이
- 활성 시간당 LOC

### Step 5: 커밋 유형 분석

관례적 커밋 접두사(feat/fix/refactor/test/chore/docs)로 분류합니다. 백분율 바로 표시:

```
feat:     20  (40%)  ████████████████████
fix:      27  (54%)  ███████████████████████████
refactor:  2  ( 4%)  ██
```

수정 비율이 50%를 초과하면 플래그를 달 — 이는 리뷰 갭이 있을 수 있는 "빠르게 출시, 빠르게 수정" 패턴을 나타냅니다.

### Step 6: 핫스팟 분석

가장 많이 변경된 파일 상위 10개 표시. 플래그:
- 5번 이상 변경된 파일 (변동 핫스팟)
- 핫스팟 목록에서 테스트 파일 vs 프로덕션 파일
- VERSION/CHANGELOG 빈도 (버전 규율 지표)

### Step 7: PR 크기 분포

커밋 diff에서 PR 크기를 추정하고 분류합니다:
- **소형** (<100 LOC)
- **중형** (100-500 LOC)
- **대형** (500-1500 LOC)
- **초대형** (1500+ LOC) — 파일 수와 함께 플래그

### Step 8: 집중도 점수 + 이번 주의 배포

**집중도 점수:** 단일 가장 많이 변경된 최상위 디렉토리(예: `app/services/`, `app/views/`)를 건드리는 커밋 비율을 계산합니다. 높은 점수 = 더 깊이 집중된 작업. 낮은 점수 = 분산된 컨텍스트 전환. 다음과 같이 보고: "집중도 점수: 62% (app/services/)"

**이번 주의 배포:** 기간 내 단일 최고 LOC PR을 자동 식별합니다. 강조 표시:
- PR 번호와 제목
- 변경된 LOC
- 왜 중요한지 (커밋 메시지와 터치된 파일에서 추론)

### Step 9: 팀원 분석

각 기여자(현재 사용자 포함)에 대해 계산합니다:

1. **커밋과 LOC** — 총 커밋, 삽입, 삭제, 순 LOC
2. **집중 영역** — 가장 많이 건드린 디렉토리/파일 (상위 3개)
3. **커밋 유형 혼합** — 개인별 feat/fix/refactor/test 분석
4. **세션 패턴** — 코딩 시간 (피크 시간), 세션 수
5. **테스트 규율** — 개인별 테스트 LOC 비율
6. **가장 큰 배포** — 기간 내 단일 최고 영향 커밋 또는 PR

**현재 사용자("당신")에 대해:** 이 섹션이 가장 심층적인 처리를 받습니다. 솔로 회고의 모든 세부 사항 포함 — 세션 분석, 시간 패턴, 집중도 점수. 1인칭으로 표현: "당신의 피크 시간...", "당신의 가장 큰 배포..."

**각 팀원에 대해:** 작업한 내용과 패턴을 커버하는 2-3문장 작성. 그런 다음:

- **칭찬** (1-2개 구체적인 것): 실제 커밋에 근거합니다. "잘 했어요"가 아닌 — 정확히 무엇이 좋았는지 말합니다. 예시: "집중된 3번의 세션에서 45% 테스트 커버리지로 전체 auth 미들웨어를 재작성했습니다", "모든 PR이 200 LOC 이하 — 규율 있는 분해."
- **성장 기회** (1개 구체적인 것): 비판이 아닌 레벨업 제안으로 표현합니다. 실제 데이터에 근거합니다. 예시: "이번 주 테스트 비율이 12% — 더 많은 기능이 추가되기 전에 결제 모듈에 테스트 커버리지를 추가하면 효과적일 것", "동일 파일에 5개의 수정 커밋은 원본 PR에서 리뷰를 더 했더라면 하는 것을 시사합니다."

**기여자가 한 명(솔로 저장소)인 경우:** 팀 분석 건너뛰고 이전과 같이 진행합니다 — 회고는 개인적입니다.

**Co-Authored-By 트레일러가 있는 경우:** 커밋 메시지에서 `Co-Authored-By:` 줄을 파싱합니다. 해당 작성자에게 주 작성자와 함께 커밋 크레딧을 부여합니다. AI 공동 작성자(예: `noreply@anthropic.com`)는 기록하지만 팀원으로 포함하지 않습니다 — 대신 "AI 지원 커밋"을 별도 메트릭으로 추적합니다.

### Step 10: 주간 추세 (기간이 14일 이상인 경우)

시간 기간이 14일 이상이면 주 단위로 분할하고 추세 표시:
- 주당 커밋 (총계 및 작성자별)
- 주당 LOC
- 주당 테스트 비율
- 주당 수정 비율
- 주당 세션 수

### Step 11: 연속 기록 추적

오늘부터 돌아가며 origin/main에 최소 1개 커밋이 있는 연속 일수를 계산합니다. 팀 연속 기록과 개인 연속 기록 모두 추적합니다:

```bash
# 팀 연속 기록: 모든 고유 커밋 날짜 (태평양 시간) — 하드 컷오프 없음
TZ=America/Los_Angeles git log origin/main --format="%ad" --date=format:"%Y-%m-%d" | sort -u

# 개인 연속 기록: 현재 사용자의 커밋만
TZ=America/Los_Angeles git log origin/main --author="<user_name>" --format="%ad" --date=format:"%Y-%m-%d" | sort -u
```

오늘부터 역방향으로 계산 — 최소 하나의 커밋이 있는 연속 일수? 전체 히스토리를 쿼리하여 모든 길이의 연속 기록이 정확하게 보고됩니다. 둘 다 표시:
- "팀 배포 연속 기록: 47 연속일"
- "당신의 배포 연속 기록: 32 연속일"

### Step 12: 히스토리 로드 및 비교

새 스냅샷을 저장하기 전에 이전 회고 히스토리를 확인합니다:

```bash
ls -t .context/retros/*.json 2>/dev/null
```

**이전 회고가 있는 경우:** Read 도구를 사용하여 가장 최근 것을 로드합니다. 주요 메트릭의 변화를 계산하고 **지난 회고 대비 추세** 섹션을 포함합니다:
```
                    지난번      이번        변화
테스트 비율:         22%    →    41%         ↑19pp
세션:               10     →    14          ↑4
LOC/시간:           200    →    350         ↑75%
수정 비율:           54%    →    30%         ↓24pp (개선)
커밋:               32     →    47          ↑47%
딥 세션:             3      →    5           ↑2
```

**이전 회고가 없는 경우:** 비교 섹션을 건너뛰고 다음을 추가합니다: "첫 번째 회고가 기록됐습니다 — 다음 주에 다시 실행하면 추세를 볼 수 있습니다."

### Step 13: 회고 히스토리 저장

모든 메트릭 계산(연속 기록 포함) 및 비교를 위한 이전 히스토리 로드 후 JSON 스냅샷을 저장합니다:

```bash
mkdir -p .context/retros
```

오늘의 다음 시퀀스 번호 결정:

```bash
today=$(TZ=America/Los_Angeles date +%Y-%m-%d)
existing=$(ls .context/retros/${today}-*.json 2>/dev/null | wc -l | tr -d ' ')
next=$((existing + 1))
# .context/retros/${today}-${next}.json으로 저장
```

### 시간 및 세션 패턴
(Step 3-4에서)

팀 전체 패턴이 의미하는 바를 해석하는 내러티브:
- 가장 생산적인 시간이 언제이고 무엇이 이를 이끄는지

### 배포 속도
(Step 5-7에서)

커버하는 내러티브:
- 커밋 유형 혼합과 그것이 드러내는 것
- PR 크기 규율 (PR이 작게 유지되고 있나?)
- 수정 체인 감지 (동일 서브시스템에 대한 수정 커밋 시퀀스)
- 버전 업데이트 규율

### 코드 품질 신호
- 테스트 LOC 비율 추세
- 핫스팟 분석 (같은 파일이 계속 변경되고 있나?)
- 분할해야 했던 초대형 PR

### 집중도 및 하이라이트
(Step 8에서)
- 해석과 함께 집중도 점수
- 이번 주의 배포 강조 표시

### 당신의 주간 (개인 심층 분석)
(Step 9에서, 현재 사용자에 대해서만)

사용자가 가장 관심 있는 섹션입니다. 포함:
- 개인별 커밋 수, LOC, 테스트 비율
- 세션 패턴 및 피크 시간
- 집중 영역
- 가장 큰 배포
- **잘 한 것** (커밋에 근거한 2-3개 구체적인 것)
- **레벨업 영역** (1-2개 구체적이고 실행 가능한 제안)

### 팀 분석
(Step 9에서, 각 팀원에 대해 — 솔로 저장소이면 건너뜀)

각 팀원(커밋 내림차순으로 정렬)에 대해 다음 섹션 작성:

#### [이름]
- **배포한 것**: 기여, 집중 영역, 커밋 패턴에 대한 2-3문장
- **칭찬**: 잘 한 1-2가지 구체적인 것, 실제 커밋에 근거. 진심이어야 합니다 — 1:1에서 실제로 말할 것을 말합니다. 구체적이고 진정성 있게.
- **성장 기회**: 1개 구체적이고 건설적인 제안. 비판이 아닌 투자로 표현합니다.

**AI 협업 노트:** 많은 커밋에 `Co-Authored-By` AI 트레일러(예: Claude, Copilot)가 있으면, AI 지원 커밋 비율을 팀 메트릭으로 언급합니다. 중립적으로 표현합니다.

### 팀 상위 3가지 성과
기간 내 팀 전체에서 가장 영향력 있는 3가지를 식별합니다. 각각:
- 무엇인지
- 누가 배포했는지
- 왜 중요한지 (제품/아키텍처 영향)

### 개선할 3가지
구체적이고, 실행 가능하고, 실제 커밋에 근거합니다. 개인적 및 팀 레벨 제안을 혼합합니다.

### 다음 주를 위한 3가지 습관
작고, 실용적이고, 현실적입니다. 각각 채택하는 데 <5분이 걸려야 합니다. 최소 하나는 팀 지향적이어야 합니다 (예: "당일에 서로의 PR을 검토합니다").

## 비교 모드

사용자가 `/retro compare` (또는 `/retro compare 14d`)를 실행할 때:

1. 현재 기간(기본 7일)에 대한 메트릭 계산
2. 겹치지 않도록 `--since`와 `--until`을 모두 사용하여 바로 이전 동일 길이 기간에 대한 메트릭 계산
3. 변화와 화살표가 있는 나란히 비교 테이블 표시
4. 가장 큰 개선과 회귀를 강조하는 간략한 내러티브 작성
5. 현재 기간 스냅샷만 `.context/retros/`에 저장; 이전 기간 메트릭은 저장하지 않음

## 톤

- 용기를 주지만 솔직하게, 달래지 않습니다
- 구체적이고 구체적 — 항상 실제 커밋/코드에 근거합니다
- 일반적인 칭찬("잘 했어요!") 건너뜁니다 — 무엇이 좋았고 왜인지 정확히 말합니다
- 개선사항을 레벨업으로 표현합니다, 비판이 아닌
- **칭찬은 1:1에서 실제로 말할 것처럼** — 구체적이고, 얻어진 것이고, 진심
- **성장 제안은 투자 조언처럼** — "이것이 당신의 시간을 쏟을 가치가 있는 이유는..." 이지 "당신이 ~에 실패했다..."가 아닌
- 팀원을 서로 부정적으로 비교하지 않습니다. 각 사람의 섹션은 독립적입니다.
- 총 출력은 약 3000-4500단어 (팀 섹션을 수용하기 위해 약간 더 길게)
- 데이터에는 마크다운 테이블과 코드 블록, 내러티브에는 산문 사용
- 대화에 직접 출력 — 파일시스템에 쓰지 않습니다 (`.context/retros/` JSON 스냅샷 제외)

## 중요 규칙

- **모든 내러티브 출력은 대화에서 사용자에게 직접** 갑니다. 작성되는 유일한 파일은 `.context/retros/` JSON 스냅샷입니다.
- 모든 git 쿼리에 `origin/main`을 사용합니다 (오래된 상태일 수 있는 로컬 main 사용 안 함)
- 표시를 위해 모든 타임스탬프를 태평양 시간으로 변환합니다 (`TZ=America/Los_Angeles` 사용)
- 기간에 커밋이 없으면 그렇다고 말하고 다른 기간을 제안합니다
- LOC/시간을 가장 가까운 50으로 반올림합니다
- 병합 커밋을 PR 경계로 취급합니다
- CLAUDE.md나 다른 문서를 읽지 않습니다 — 이 스킬은 독립적입니다
- 첫 번째 실행(이전 회고 없음)은 비교 섹션을 우아하게 건너뜁니다
