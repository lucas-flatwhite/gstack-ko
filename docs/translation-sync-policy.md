# 번역 싱크 원칙

## 목적

`gstack-ko`는 `garrytan/gstack`의 한국어 문서와 스킬 설명을 제공하는 저장소다. 이 저장소의 목적은 실행 가능한 독립 포크를 유지하는 것이 아니라, upstream의 문서와 워크플로우를 한국어 사용자에게 이해하기 쉽게 제공하는 것이다. 한 줄로 말하면, `gstack-ko`는 `문서/스킬 저장소`다.

## 저장소 정체성

- 이 저장소는 `문서/스킬 전용 저장소`다.
- 실행 코드, 테스트, 빌드 체계의 진실 원천은 upstream `garrytan/gstack`에 둔다.
- upstream와 동일한 기능 포크나 독립 배포판을 목표로 하지 않는다.

## 유지 대상

- 루트 문서 (`README.md`, `AGENTS.md`, `CONTRIBUTING.md`, `CHANGELOG.md`, `CLAUDE.md` 등)
- 루트 `SKILL.md`
- 각 스킬 디렉터리의 `SKILL.md`
- supporting reference docs
- `docs/` 아래의 정책 및 운영 문서

## 반입 금지 대상

다음 항목은 upstream에 존재하더라도 이 저장소에 반입하지 않는다.

- 실행 코드와 라이브러리 소스
- 테스트 코드와 fixture
- 빌드 설정과 의존성 매니페스트
- lockfile, 바이너리, 설치 스크립트
- CI workflow와 코드 생성 체계

## 번역 규칙

- 명령어, 옵션, 코드 블록, 경로, 변수명, 고유 명칭은 번역하지 않는다.
- 설명 텍스트만 한국어로 제공한다.
- upstream 문맥을 이해하는 데 필요한 경우 링크와 짧은 설명을 덧붙이되, 코드 자체를 복제하지 않는다.

## Upstream Sync 절차

1. upstream 릴리스 또는 주요 문서 변경을 확인한다.
2. 문서와 스킬 변경만 선별한다.
3. 한국어 저장소의 범위를 벗어나는 파일은 반입하지 않는다.
4. 이미 반입된 범위 밖 파일이 있다면 sync 과정에서 제거한다.
5. `README.md`, `CHANGELOG.md`, 관련 정책 문서가 현재 범위를 정확히 설명하는지 함께 검토한다.

## PR 리뷰 기준

다음 질문에 모두 `예`라고 답할 수 있어야 한다.

- 이 변경은 문서 또는 스킬 설명의 품질을 높이는가
- 저장소의 범위를 문서/스킬 밖으로 넓히지 않는가
- upstream와 이 저장소의 역할 경계가 분명한가
- 한국어 번역 규칙을 지켰는가

다음 유형의 PR은 원칙적으로 받지 않는다. 코드 반입 PR은 원칙적으로 받지 않는다.

- 실행 코드 추가 또는 수정 PR
- 테스트, fixture, 바이너리 반입 PR
- 빌드/배포 체계 유지보수 PR

## 스킬 문서 운영 원칙

- `SKILL.md`는 한국어 reference 문서로 유지한다.
- `SKILL.ko.md` 같은 병행 파일은 두지 않는다. 이 저장소 안에서 한국어 문서의 기본 파일명은 그대로 `SKILL.md`다.
- 모든 스킬 문서를 기계적으로 반입하지 않고, 저장소 정체성에 직접 필요한 핵심 스킬만 선별 유지한다.
- 실행 코드가 없는 스킬은 문서 상단 또는 관련 위치에서 upstream 의존성을 명확히 안내한다.
- 브라우저, QA, 쿠키 import, 업그레이드, 배포처럼 실행 환경 의존성이 큰 스킬은 "이 저장소만으로는 실행되지 않는다"는 점을 숨기지 않는다.

## 완료 기준

한 번의 sync 작업은 아래를 만족할 때 완료된 것으로 본다.

- 문서와 스킬 설명 변경이 한국어로 반영되었다.
- 범위 밖 파일이 새로 반입되지 않았다.
- 저장소 소개 문서와 기여 가이드가 현재 운영 원칙과 모순되지 않는다.
