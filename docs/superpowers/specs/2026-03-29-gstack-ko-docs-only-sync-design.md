# gstack-ko 문서/스킬 전용 저장소 전환 설계

## 목적

`gstack-ko`를 원본 `gstack`의 한국어 포크가 아니라, 한국어 문서와 스킬을 제공하는 전용 저장소로 명확히 재정의한다. 이 설계의 목표는 upstream sync 비용을 낮추고, 외부 기여자도 같은 기준으로 판단할 수 있는 공개 규칙을 만드는 것이다.

## 결정 사항

### 저장소 정체성

- `gstack-ko`는 한국어 문서/스킬 저장소다.
- 이 저장소는 독립 실행 배포판이나 기능 포크를 목표로 하지 않는다.
- 실행 코드, 테스트, 빌드 체계의 진실 원천은 upstream `garrytan/gstack`에 둔다.

### 유지 범위

유지 대상:

- 루트 및 각 스킬 디렉터리의 한국어 `SKILL.md`
- 한국어 `README.md`
- 한국어 `CONTRIBUTING.md`
- 한국어 `ARCHITECTURE.md`
- 한국어 `CHANGELOG.md`
- 저장소 정책 문서와 번역 운영 문서

제외 대상:

- 실행 코드 (`browse/src`, CLI, 서버 구현 등)
- 테스트 코드와 fixture
- 빌드 설정 파일과 의존성 매니페스트
- 바이너리와 설치 스크립트
- 템플릿 생성 체계 (`*.tmpl`, 문서 생성 스크립트)

예외는 두지 않는다. 문서 설명을 위해 코드가 필요하더라도 이 저장소 안에 복제하지 않고 upstream 링크나 발췌 설명으로 해결한다.

## 정보 구조 변경

### 저장소 구조

문서/스킬 전용 구조로 단순화한다.

- 각 스킬 디렉터리는 `SKILL.md`만 유지한다.
- 문서성 루트 파일(`README.md`, `CONTRIBUTING.md`, `ARCHITECTURE.md`, `CHANGELOG.md`, `CLAUDE.md`, `TODOS.md`)만 유지한다.
- 정책 문서는 `docs/` 아래에 둔다.
- 실행 코드와 테스트 디렉터리는 제거한다.

### 새 정책 문서

공개 규칙 문서를 추가한다. 이 문서는 다음을 포함해야 한다.

- 저장소의 범위와 비범위
- upstream sync의 목적과 완료 기준
- 허용되는 변경과 거절 대상
- PR 리뷰 기준
- README/CHANGELOG 갱신 규칙
- 새 스킬 또는 문서 추가 시 따라야 하는 기준

정책 문서 파일명은 `docs/translation-sync-policy.md`로 고정한다.

## Upstream Sync 원칙

### 기본 원칙

- upstream sync는 “원본 변경을 한국어 문서/스킬에 반영하는 작업”이다.
- upstream의 코드 변경을 이 저장소에 재현하는 작업이 아니다.
- sync 결과는 항상 문서 저장소의 정체성을 더 명확하게 만들어야 한다.

### 가져오는 것

- 새 스킬의 한국어 문서
- 기존 스킬 문구 변경
- README, 기여 가이드, 아키텍처 문서, 변경 이력 같은 문서 변경
- 문서 이해를 위해 필요한 명령어명, 옵션명, 경로, 고유 명칭

### 가져오지 않는 것

- 실행 코드 추가/수정
- 테스트 추가/수정
- 빌드/릴리스 인프라
- 바이너리, lockfile, package manager 설정
- 생성용 템플릿이나 생성 스크립트

### sync 시 처리 규칙

1. upstream 릴리스를 확인한다.
2. 문서와 스킬 변경만 선별한다.
3. 번역 저장소 범위를 벗어나는 파일은 반입하지 않는다.
4. README와 정책 문서가 현재 범위를 정확히 설명하는지 함께 검토한다.
5. CHANGELOG에는 한국어 저장소 기준으로 의미 있는 변화만 기록한다.

## README 설계

README는 첫 화면에서 다음 사실을 분명히 보여줘야 한다.

- `gstack-ko`는 한국어 문서/스킬 저장소다.
- 실행 가능한 전체 기능 세트는 upstream `gstack`가 제공한다.
- 브라우저/QA/쿠키 import 같은 실행 기능은 이 저장소가 아니라 upstream를 따라야 한다.
- 이 저장소의 가치는 한국어 설명, 한국어 스킬 문서, 한국어 운영 가이드에 있다.

기존 README의 기능 소개는 유지하되, “무엇이 여기 있고 무엇이 upstream에 있는지”를 더 앞쪽에서 명확히 구분한다.

## CONTRIBUTING 설계

기여 가이드는 외부 기여자에게 허용 범위를 명확히 알려야 한다.

- 환영하는 기여: 번역 개선, 오탈자 수정, 문서 구조 개선, 스킬 설명 업데이트, 정책 문서 보강
- 제한하는 기여: 실행 코드, 테스트, 빌드 시스템, 바이너리 반입
- sync PR에는 정책 문서 체크리스트를 통과하도록 요구한다.
- 번역 규칙은 “명령어, 옵션, 코드 블록, 경로, 변수명은 번역하지 않는다”를 기본 원칙으로 유지한다.

## 구현 범위

이번 변경에서 수행할 작업은 다음과 같다.

- 새 정책 문서 추가
- `README.md`를 문서/스킬 전용 저장소 방향에 맞게 개정
- `CONTRIBUTING.md`를 문서 기여 중심으로 개정
- 저장소 내 실행 코드, 테스트, 빌드 관련 파일 제거
- 필요 시 `ARCHITECTURE.md`, `CHANGELOG.md`, `TODOS.md`, `CLAUDE.md` 문구를 범위에 맞게 보정

## 검증 기준

변경 후 다음을 만족해야 한다.

- 저장소 트리에서 프로그램 코드와 테스트가 제거되어 있다.
- README만 읽어도 이 저장소의 범위와 한계를 이해할 수 있다.
- CONTRIBUTING만 읽어도 어떤 PR이 허용되는지 판단할 수 있다.
- 정책 문서만 읽어도 다음 upstream sync를 같은 기준으로 수행할 수 있다.

## 리스크와 대응

### 리스크 1: 일부 사용자가 독립 실행 저장소로 오해

대응:

- README 서두에 비포함 범위를 명시한다.
- 실행 기능은 upstream로 안내한다.

### 리스크 2: 외부 기여자가 코드 반입 PR을 열 가능성

대응:

- 정책 문서와 CONTRIBUTING에 거절 기준을 명시한다.
- PR 템플릿이 없다면 향후 추가를 검토한다.

### 리스크 3: upstream 문서가 코드 설명에 강하게 의존

대응:

- 코드 자체를 들여오지 않고, 필요한 경우 개념 설명과 링크로 대체한다.
- 설명이 과도하게 길어지면 “upstream 참고”를 정식 원칙으로 둔다.
